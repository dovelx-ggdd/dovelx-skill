# 专业审查团队 (Review Team)

> **输出语言规则**：所有审查报告**必须使用中文**撰写。

## 角色定位

协调 3 个专项审查 Agent 并行执行，覆盖所有审查维度，最终汇总成一份结构化报告。

## 何时激活

- 用户提交需求文档、PRD、用户故事
- 用户提交技术设计文档、API 设计、数据库 Schema
- 用户提交代码（新功能 or 重构）
- 用户说"帮我审查"、"review 一下"、"检查这个"

---

## 审查团队（3 个并行 Agent）

| Agent | 配置文件 | 职责 | 涵盖维度 |
|-------|---------|------|---------|
| **Agent A** | `agents/spec-reviewer.md` | 检查"做对的事"与"设计是否合理" | 需求完整性、设计合理性、重构上下文 |
| **Agent B** | `agents/code-standards-reviewer.md` | 检查代码本身的质量 | 命名规范、结构规范、边界值与异常覆盖 |
| **Agent C** | `agents/security-performance-reviewer.md` | 检查非功能质量 | 安全性（OWASP）、性能与资源使用 |

> 各 Agent 的详细审查清单见对应配置文件。

---

## 执行流程

### Step 1：识别提交物与场景

| 提交物类型 | Agent A | Agent B | Agent C |
|-----------|---------|---------|---------|
| 需求文档 / PRD | ✅（需求为主） | ✅（结构规范） | ✅ |
| 技术设计文档 | ✅（设计为主） | ✅ | ✅ |
| 新功能代码 | ✅（设计合理性） | ✅ | ✅ |
| 重构代码 | ✅（**含重构上下文**） | ✅ | ✅ |

### Step 2：并行派发 3 个 Agent

使用 Agent tool 同时启动 3 个子 Agent，每个 Agent 独立审查，互不干扰。  
每个 Agent 的 prompt 中包含：提交物内容 + 对应审查清单（来自 agents/ 配置文件）。

---

## Step 3：汇总报告

收集 3 个 Agent 结果，汇总后保存审查报告。

完整报告结构参见模板：[review-template.md](review-template.md)

主要章节：总体结论（🟢/🟡/🔴）、Agent A/B/C 各自评级与问题清单、优先级汇总表、亮点、修复验证清单。

### 文档保存规程

保存报告时**必须**按以下步骤操作：

1. **确定功能模块名**：根据当前审查对象确定（英文小写、连字符分隔）
2. **检查目录是否存在**：判断 `.claude/doc/<功能名>/` 是否存在
   - 已存在 → 直接使用，**不重新创建**
   - 不存在 → 创建该目录
3. **确定版本号**：查找目标目录下 `code-review-<日期>-v*.md` 文件
   - 已有同类型文件 → 取最大版本号 +1
   - 无同类型文件 → 版本号从 `v1` 开始
4. **保存**：`.claude/doc/<功能名>/code-review-<YYYY-MM-DD>-v<N>.md`

## 严重等级

| 等级 | 含义 | 处理 |
|------|------|------|
| 🔴 CRITICAL | 安全漏洞 / 数据丢失 / 严重逻辑错误 | **阻止合并，必须修复** |
| 🟠 HIGH | Bug / 重大性能 / 需求缺失 | **建议修复后合并** |
| 🟡 MEDIUM | 可维护性 / 边界遗漏 / 命名不清 | **考虑修复** |
| 🟢 LOW | 风格建议 / 细节优化 | **可选** |
