# Dev Design 工作流详细参考

## Phase 1: Requirement Discovery — 详细指引

### 问询策略

优先使用 AskQuestion 工具进行结构化收集，每轮不超过 3-4 个问题，避免信息过载。

**第一轮（目标与场景）：**
1. 这个需求要解决什么核心问题？
2. 典型的用户使用场景是什么？
3. 是否有已存在的相关功能可以参考？

**第二轮（约束与标准）：**
1. 有哪些技术或业务约束？
2. 验收标准是什么？如何判断需求完成？
3. 是否涉及外部系统集成？

**第三轮（补充，按需）：**
1. 数据量和性能要求？
2. 是否有时间要求或里程碑？
3. 是否需要考虑权限控制？

### 输出格式

```markdown
## 需求简报

**需求名称**：[名称]
**核心目标**：[一句话描述]
**用户场景**：
- 场景 1：...
- 场景 2：...
**约束条件**：...
**验收标准**：
- [ ] ...
**关联功能/模块**：...
```

## Phase 2: Requirement Analysis — 详细指引

### 分析检查清单

- [ ] 需求是否可以独立交付？
- [ ] 是否存在隐含的子需求？
- [ ] 数据流是否完整？（输入 → 处理 → 输出）
- [ ] 错误/异常场景是否覆盖？
- [ ] 是否与现有功能冲突？
- [ ] 是否需要数据库变更？
- [ ] 是否需要新的 API 端点？
- [ ] 是否需要权限/角色控制？

### 代码库探索

使用以下工具快速了解影响范围：
- `Grep` — 精确查找类名、方法名、表名、关键字
- `Glob` — 查找相关文件结构和模式
- `Agent (Explore)` — 语义级别的代码库探索
- 读取项目现有核心模块了解架构约定（如有 `CLAUDE.md` 优先读取）

## Phase 4: Challenge Review — 详细指引

### Review Agent Prompt 模板

```
你是一位资深架构师，请对以下实施计划进行严格评审：

[插入 Draft Plan]

评审维度：
1. 完整性：计划是否覆盖了所有验收标准？
2. 可行性：是否存在技术障碍或不合理假设？
3. 一致性：是否与现有架构和编码规范一致？
4. 边界条件：缺少哪些异常/边界场景处理？
5. 替代方案：是否存在更简单或更健壮的实现方式？

请按以下格式输出：
- 问题列表（严重性：高/中/低）
- 替代方案（如有，附优劣对比）
- 总体建议
```

### 方案对比格式

```markdown
## 方案对比

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 描述 | ... | ... | ... |
| 优点 | ... | ... | ... |
| 缺点 | ... | ... | ... |
| 复杂度 | L/M/H | L/M/H | L/M/H |
| 风险 | ... | ... | ... |
| 推荐度 | ⭐⭐⭐ | ⭐⭐ | ⭐ |
```

## Phase 5: Feasibility Review — 详细指引

### Review Agent Prompt 模板

```
你是一位技术评审专家，请对以下选定方案进行可行性深度评审：

[插入选定方案详情]

评审重点：
1. 技术可行性：选定技术栈/框架是否支持？
2. 边界条件：所有异常场景是否有处理策略？
3. API 契约：接口设计是否完整、向后兼容？
4. 数据模型：表结构是否合理、是否需要迁移脚本？
5. 性能：是否存在性能瓶颈？
6. 安全：是否存在安全隐患？

请输出：通过/不通过 + 详细评审意见
```

## Phase 7: Design Review — 详细指引

### Review 循环规则

1. 最多迭代 3 轮
2. 每轮聚焦于上一轮发现的问题
3. 如果第 3 轮仍有高严重性问题，标记为已知风险并记录到设计文档

### Review Agent Prompt 模板

```
请评审以下设计文档的完整性、技术准确性和实施可行性：

[插入 design.md 内容]

评审检查清单：
- [ ] 概述是否清晰完整？
- [ ] 验收标准是否可验证？
- [ ] 数据模型是否合理？
- [ ] API 设计是否符合 RESTful 规范？
- [ ] 实施计划是否可执行？
- [ ] 风险是否已识别并有缓解措施？
- [ ] 测试策略是否充分？

输出：问题列表 + 修改建议
```

## Phase 9: Verification — 详细指引

### 自测检查清单

- [ ] 所有新增/修改的文件无 linter 错误（运行项目 lint 命令验证）
- [ ] 函数/方法职责单一，长度 < 50 行
- [ ] 错误处理完整，不吞异常
- [ ] 所有用户输入已校验
- [ ] 无硬编码密钥或魔法值
- [ ] API 路径符合项目已有规范（REST 或 RPC）
- [ ] 数据库查询使用参数化，无 SQL 注入风险
- [ ] 新增接口有对应测试覆盖（目标 80%+）

### 需求回归验证格式

```markdown
## 需求回归验证

| # | 验收标准 | 状态 | 验证方式 | 备注 |
|---|----------|------|----------|------|
| 1 | ... | ✅/❌ | 代码检查/测试 | ... |

**结论**：[全部通过 / X 项未通过，需处理：...]
```
