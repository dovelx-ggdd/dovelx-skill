# 开发团队编排 (Dev Team Orchestrator)

> **输出语言规则**：所有生成的文档**必须使用中文**撰写。

## 角色定位

协调三个专属技能，按正确顺序执行开发流程：

```
dovelx-requirements  →  dovelx-tech-design  →  开发实现  →  dovelx-code-review
      需求分析                技术设计                编码              代码审查
      ↓ PRD                  ↓ 设计文档               ↓ 代码            ↓ 审查报告
```

## 何时激活

| 用户说... | 从哪里开始 |
|-----------|-----------|
| "开始做 X 功能" | 需求分析 → 完整流程 |
| "设计 X 方案" | 技术设计 |
| "审查这段代码" | 代码审查 |
| "走完整流程" | 需求分析 → 技术设计 → 开发 → 代码审查 |

## 四个阶段

### 阶段 1：需求分析

**执行方式**：派发 `agents/requirements-analyst.md` 子 Agent  
**产出**：PRD → `.claude/doc/<功能名>/prd-<日期>-v1.md`  
**门控**：用户确认 PRD 后才进入下一阶段

> 也可直接调用 `/dovelx-requirements` 由用户主导交互式需求分析。

### 阶段 2：技术设计

**执行方式**：派发 `agents/tech-designer.md` 子 Agent，输入为阶段 1 的 PRD 路径  
**产出**：技术设计文档 → `.claude/doc/<功能名>/design-<日期>-v<N>.md`  
**门控**：用户确认技术方案后才开始实现

> 也可直接调用 `/dovelx-tech-design`。

### 阶段 3：开发实现

**执行方式**：主 Claude 实例按技术设计文档逐项实现  
按设计文档的模块划分顺序编码，遵循项目既有规范，同步编写测试。

### 阶段 4：代码审查

**执行方式**：派发 `agents/code-reviewer.md` 子 Agent，输入为变更文件列表  
**产出**：审查报告 → `.claude/doc/<功能名>/code-review-<日期>-v1.md`  
**门控**：CRITICAL 问题全部修复后才能合并

> 也可直接调用 `/dovelx-code-review`。

## 单阶段快速入口

```
/dovelx-requirements  — 只做需求分析
/dovelx-tech-design   — 只做技术设计
/dovelx-code-review   — 只做代码审查
/dovelx-dev-team      — 完整流程编排
```

## 文档归档

```
.claude/doc/<功能名>/
├── design-<日期>-v1.md       # PRD（需求阶段）
├── design-<日期>-v2.md       # 技术方案（设计阶段，已有 v1 则顺延）
└── code-review-<日期>-v1.md  # 审查报告（审查阶段）
```

### 文档保存规程（各阶段均须遵守）

每个阶段输出文档前**必须**执行：

1. **检查目录**：判断 `.claude/doc/<功能名>/` 是否存在
   - 已存在 → 直接使用，**不重新创建**
   - 不存在 → 创建目录
2. **确定版本号**：查找目标目录下同类型同日期文件（`<类型>-<日期>-v*.md`）
   - 已有同类型文件 → 取最大版本号 +1
   - 无同类型文件 → 从 `v1` 开始
3. **保存**：`.claude/doc/<功能名>/<文档类型>-<YYYY-MM-DD>-v<N>.md`

## 进度追踪

每个阶段开始时创建 Task，完成时标记完成：
- `[ ] 需求分析` → 完成后注明 PRD 路径
- `[ ] 技术设计` → 完成后注明设计文档路径
- `[ ] 开发实现` → 完成后列出主要变更文件
- `[ ] 代码审查` → 完成后注明报告路径和最终结论
